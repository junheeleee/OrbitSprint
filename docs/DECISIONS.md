# Decisions

## Documentation

두 프로젝트 모두 빠르게 바뀌는 게임 프로젝트이므로, 기록의 기준은 Notion보다 GitHub와 repo 내부 문서로 둔다.

결정:

- 코드/버그/릴리즈: GitHub
- 요구사항/결정/작업 로그: `docs/`
- 보기 좋은 외부 기획서: 필요할 때 Notion으로 복제
- 실시간 팀 커뮤니케이션: 팀원이 생기기 전까지 Slack은 보류

이유:

- 커밋과 문서가 같은 히스토리에 남는다.
- “왜 이렇게 바꿨는지”를 코드 근처에서 추적할 수 있다.
- 앱스토어 출시 전 QA/릴리즈 노트로 변환하기 쉽다.

## Lumen Run

결정:

- 제목과 프로젝트명은 `Lumen Run`으로 통일
- 궤도 조작은 복잡한 투버튼보다 원버튼 방식 유지
- 3단계 궤도 이동은 `1 -> 2 -> 3 -> 2 -> 1`
- 피버 모드에서는 무적과 고속 점수 획득을 제공

이유:

- 모바일 아케이드 게임은 조작 이해가 빨라야 함
- 깊이는 조작 복잡도보다 패턴, 리스크, 피버 타이밍에서 만드는 편이 적합
- IP 기억성을 위해 제목, 음악, 이펙트, UI 톤을 통일해야 함

## Gangnam Dream

결정:

- Godot 4.6 기준으로 유지
- 이벤트와 콘텐츠는 JSON 데이터로 관리
- 시스템은 manager/autoload 중심으로 분리
- UI는 대시보드형 텍스트 RPG로 구성
- 복잡한 그래픽보다 선택, 경제 압박, 인간관계, 뉴스, 투자 시뮬레이션을 우선

이유:

- 이벤트가 수백 개로 늘어날 게임이므로 코드에 직접 박으면 유지보수가 어려움
- 로그라이크 반복 플레이는 콘텐츠 조합과 조건 시스템이 핵심
- 한국 현대사회 풍자는 텍스트, 숫자, 뉴스 헤드라인, 선택 결과가 잘 맞음
